Treat statements using the following evidence hierarchy:

- Assumption: a reasonable choice needed because the requirement is silent.
- Constraint: a stated or unavoidable limitation.

Never present an assumption or preferred design as if it were an explicit business requirement. State all material assumptions clearly in the assumptions table. # Solution Architecture Design Document (SADD) — [SOLUTION_NAME]

## 1. Executive Summary

- Summarize the business problem, target users, proposed Salesforce solution, external systems, and architectural outcome.
- State precisely which requirement document is covered and what is excluded.
- Keep the summary concise but specific to the new business process.

## 2. Requirement Coverage

### 2.1 Requirement Coverage Map

- Include a PlantUML map from requirement groups to solution capabilities.
- Follow it with a detailed Markdown traceability table.
- Link each requirement to the section where its design is explained.
- Include functional, data, integration, identity, security, reporting, migration, volume, and non-functional requirements where applicable.

## 3. Scope, Assumptions, Constraints, and Open Questions

### 3.1 In Scope
### 3.2 Out of Scope
### 3.3 Assumptions
### 3.4 Constraints
### 3.5 Open Questions / Validation Items

- Give assumptions stable IDs such as A-01 and constraints IDs such as C-01.
- Make scope boundaries unambiguous.

## 4. Salesforce Product and Capability Selection

### 4.1 Capability Selection
### 4.2 License and Commercial Validation
### 4.3 Build-versus-Buy or Standard-versus-Custom Decisions, when relevant

- Map requirement areas to recommended Salesforce clouds/features and justified third-party or middleware capabilities.
- Include rationale and meaningful alternatives/trade-offs.
- Separate mandatory baseline, conditional, optional, and commercial-validation items.

## 5. Target Solution Architecture

### 5.1 Architecture Overview / System Landscape
### 5.2 Layered Architecture
### 5.3 Context or Container View, when it adds clarity
### 5.4 Data Flow or Trust-Boundary View, when relevant

- Explain actors, channels, Salesforce clouds, platform services, external systems, identity providers, middleware, and principal data stores.
- The prose must explain what the diagrams mean, not merely repeat labels.

## 6. Business Process Architecture

### 6.1 Persona and User Story Map
### 6.2 End-to-End Business Process Overview
### 6.x One detailed subsection for every primary business process
### 6.x Shared intake, validation, matching, triage, routing, approval, escalation, fulfillment, notification, and closure subprocesses as applicable
### 6.x Exception and alternate-path process where material

For each primary process:

- Include a PlantUML activity, use-case, BPMN-like activity, or sequence diagram.
- Include a table mapping each business step to the Salesforce implementation.
- Preserve explicit decisions, exception paths, manual fallbacks, approvals, SLAs, and closure conditions.
- Do not combine materially different business journeys into an unreadable single diagram.

Write user stories in the form: “As a [persona], I want [capability], so that [business outcome].” Cover all personas and major responsibilities.

## 7. Application Architecture

### 7.1 Application Layer Overview
### 7.2 Service / Component Layer Design
### 7.3 Important Service Interaction Sequence(s)
### 7.4 Error Handling Strategy
### 7.5 Configurable Business Rules

- Define canonical service/component responsibilities without implying custom Apex is mandatory.
- Identify Flow, approval, orchestration, LWC, Apex, async jobs, platform events, custom metadata, APIs, and standard Salesforce services only where justified.
- Separate logical dependencies from runtime call sequences.
- Explain fault behavior, idempotency, rollback boundaries, bulk safety, and support visibility.

## 8. Data Architecture

### 8.1 Entity Relationship Diagram
### 8.2 Core Data Model
### 8.3 Recommended Objects and Fields
### 8.4 Record Type / Type / Category Design where applicable
### 8.x Lifecycle State Diagrams for every major transactional entity
### 8.x Data Ownership, Quality, Retention, and Archiving

- Prefer standard objects where appropriate and justify custom objects.
- Include key standard and custom fields needed by processes, routing, integration, audit, reporting, migration, and security.
- Make cardinalities explicit in the ERD.
- Ensure every object and field referenced elsewhere exists here, and every material ERD entity has an explained purpose.
- Ensure lifecycle status values exactly match process diagrams, automation, reporting, and closure controls.

## 9. Integration Architecture

### 9.1 Integration Pattern Matrix
### 9.x One subsection and sequence/data-flow diagram for every material integration
### 9.x Retry, Idempotency, Error Handling, and Reconciliation
### 9.x Channel or Interface Integrations
### 9.x Integration Security and Credential Model

For every integration, document:

- Source and target.
- System of record.
- Direction and trigger.
- Synchronous, asynchronous, batch, event-driven, or virtualized pattern.
- Expected contract/protocol and payload ownership.
- Authentication and authorization.
- Timeout, retry, backoff, dead-letter/manual-review behavior.
- Idempotency/correlation key and duplicate prevention.
- Logging, alerting, reconciliation, replay, and data minimization.
- Whether middleware is required, optional, or unnecessary, with rationale.

## 10. Security, Access, Identity, and Compliance

### 10.1 Role-Based Access and Visibility Model
### 10.2 Persona Access Matrix
### 10.3 Authentication / SSO / External Identity Flow as applicable
### 10.4 Security Controls
### 10.5 Privacy, Audit, Segregation of Duty, and Compliance Considerations

- Model internal, external, partner, administrator, integration, and migration personas only when supported by the new requirement or an explicit assumption.
- Explicitly protect sensitive data and cross-business-unit boundaries.

## 11. Data Migration, Volume, and Lifecycle Strategy

### 11.1 Data Volume and Storage Strategy
### 11.2 Migration Architecture and Execution Plan
### 11.3 Large Data Volume Design

### 12.2 Performance / Outcome Evaluation Model
### 12.x Persona-Specific Dashboards and KPIs

## 13. Non-Functional Requirements

- Cover scalability, performance, availability, reliability, recoverability, security, privacy, compliance, flexibility, maintainability, observability, accessibility, localization, deployment, and supportability as relevant.
- State measurable values only when sourced or explicitly marked as a proposed target requiring validation.

## 14. Risks and Mitigation

- Include risk, cause/trigger, impact, likelihood or priority where useful, mitigation, contingency, and owner/TBD where appropriate.
- Make risks specific to the new requirement and proposed architecture.

## 15. Architectural Decision Records

- Use stable IDs such as ADR-001.
- For each material decision include Decision, Requirement/Context, Rationale, Alternatives Considered, and Consequences/Trade-offs.
- Include decisions about the primary Salesforce data model, products, automation approach, integration pattern, identity, security/sharing, files/storage, migration, and any major customization.

Add further sections if required, for example Deployment Architecture, Environment Strategy, DevSecOps/CI-CD, Testing Strategy, Disaster Recovery, or Governance. Do not omit a topic explicitly requested by the new requirement merely because it is absent from the reference SADD.
